# Design Decisions

[README](../README.md) の Design Decisions を圧縮せず、判断の全文をここに置く。「なぜそうしたか」だけでなく「なぜ別案にしなかったか」を辿れるようにする。

## pip（PyPI）配布を採った

module-guide の既定は clone リファレンスだが、`ops_dynamic` が本番でコアを組み込む実利用者としてすでに存在するため「組み込み型 → レジストリ配布必須」に該当する。pip 化は PYTHONPATH 参照をバージョン固定依存に置き換える実運用の改善であり、かつ「wheel に答えが存在しない」ことが構造的分離の証明になる。versioning・後方互換・英語 doc の継続コストは承知の上で払う。名前は `bt-dynamic` / `bt_dynamic` に統一（`dynamic-regime-bt` 案は破棄）。

## 実データの自取得を標準の使い方とする

利用者が Dukascopy 等から自分で取得した実データ（1h足で十分）で回すのが標準。禁じられているのはデータの塊の再配布であって取得・利用ではないため、リポには取得手順と変換スクリプトを置き、データファイルはコミットしない。同梱の合成データ（シード付き乱数ウォーク）は「テストとデモが誰の環境でも再現できる」ための素材に位置づけを限定する。Saxo 等プラットフォーム由来のデータはリポに入れない。

## 差し替えは CLI 引数とファイルで行う。対話ウィザードにしない

「インジケーターとパラメータを差し替えて、結果を見て、また回す」がツールの本質価値。パラメータは `--param key=value` の引数上書き（試行のたびに JSON 編集を強制しない）、インジケーターは `IndicatorSet` を公開する Python ファイルの `--indicators` 注入とする。戦略追加は dotfiles 版の「1戦略1ディレクトリ」を踏襲し、examples をコピーして indicators.py と config.json を差し替える方式を README でアドバイスする。利用者は Python を書ける層なので、対話的な追加ウィザードは機能でなく摩擦になる——実装はファイルでやればよい。

## 動的レジーム切替を採った

静的バックテストは相場環境の変化に弱く共倒れする。相場を短窓で分類し局面ごとに戦略を切り替えることで、過学習を避ける設計判断。比較検証で自分の仮説を潰しにいく姿勢もこの方針の一部（比較の手段は後述の `--json` 汎用化を参照）。

## エッジは枠組みではなく「対応表の中身」。パラメータも対応表も設定 JSON で外部注入する

ADX/ATR/RSI も9セルも動的閾値もクオンツの一般概念で、公開してもエッジは漏れない。唯一エッジ形状を示すのは `regime_strategy`（どの局面で follow/flip が効くか＝答え）。これは中立パラメータでも形が残るため、パラメータだけでなく**対応表自体も外部注入**にし、リポには中立／説明用ダミーのみ置く。線引きは「問いと手法は出す、答えは出さない」。

注入の形式は**設定 JSON 一本**（環境変数 `$BT_DYNAMIC_CONFIG` はファイルパスの指定のみ。初期案にあったパラメータ個別の env 注入は、設定の散逸を招くため廃止した）。本番値をコードに焼き込まず外部注入にすることで、リポにも wheel にもデモにも**そもそも存在しない**状態を作る——守秘を「隠蔽」ではなく「構造的分離」で実現し、再現性（誰でも clone して動かせる）と両立させる。

## コアは戦略を import しない（依存逆転）

dotfiles 版はエンジン（core/backtest.py）が戦略側（trend/rules.py の対応表）を import しており、コードを配ればエッジも付いてくる構造だった。OSS 化にあたり依存を逆転し、対応表は Config として引数で受け取る・import 時のグローバル設定ロードを廃止する・CLI がモジュール定数を書き換える方式をやめ `run_day()` に全パラメータを引数で渡す、に改めた。「wheel に答えが存在しない」を実装レベルで成立させているのはこの逆転で、外部注入の判断と表裏一体。

## 設定キーは指標非依存の一般名にし、ADX/ATR/RSI は「デフォルト指標」に降格する

エンジンが知るのは抽象軸のみ（`ax1` = トレンド強度、`ax2` = ボラティリティ、`direction` = 中心値付き振動子）。閾値キーも `ax1_weak` / `vol_lo` / `direction_band` と一般名にし、ADX/ATR/RSI は `IndicatorSet` の差し替え可能なデフォルト実装に位置づけた。`--indicators file.py` による指標差し替えはこの抽象化が成立条件。dotfiles 版のキー名（`adx_weak` 等）からは改名が必要になるが、設定ロード時のエラーに改名ヒントを出して移行コストを吸収する。

## 比較スクリプト群は移植せず、`--json` 出力に汎用化する

dotfiles 版の compare_dynamic / compare_lot / compare_regimes は、比較したい仮説を1本ずつスクリプトに焼き込む方式で、仮説が増えるたびコードと保守が増えた。OSS 版では実行結果を `--json` で機械可読に吐き、`>> runs.jsonl` して jq 等で並べる汎用ワークフローに置き換える。任意の軸の比較が追加コードなしででき、専用スクリプトがエッジ（何を比較していたか）を示唆するリスクも消える。

## lot 概念を持たない

エンジンの成績は flat lot の pips 集計のみで、lot 重み付け（旧 compare_lot の adx_prop 等）は持たない。lot サイジングはエンジンの判定ロジックと独立した資金管理の関心事であり、足すならポジションモデルの設計判断が要る。dogfooding で必要が立った時点で Issue として扱う。

## 戦略を trend/ 1本に絞る

「単一仮説を端から端まで」というナラティブが本リポの強み。戦略を足すと「色々試す人」に戻り、エッジ形状のヒントも保守コストも増える。1本をクリーンに見せる方がポートフォリオとして強い。

## README / Zenn に投資する

リポの価値は配管コードでなく思考。文章化して初めて汎用エンジンとの差（完結性）が立つ。書くのは問いと手法まで。成績・対応表の中身・どのセルが効いたかは書かない。「答えは構造的に分離してある」と言えること自体が設計力の証明になる。

## データを同梱しない

各データソース（Dukascopy / Alpha Vantage / HistData / yfinance）はデータの塊の再配布を規約で制限する。一方コードの配布は自由。よって「取得コードを配り、データは利用者が自取得」とする。VHS で数行を映すのは復元不能な断片＝ de minimis であり再配布に当たらない。

## Demo で成績を出さない

公開する中立パラメータ × 無料1h足のバックテスト成績は、本番（専用 feed × 秘密パラメータ）と無関係。数字を実績として見せると誤読を生むため、Demo の用途は「仕組みと検証作法の実演」に固定する。
