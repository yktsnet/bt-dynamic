# Design Decisions

[README](../README.md) の Design Decisions を圧縮せず、判断の全文をここに置く。「なぜそうしたか」だけでなく「なぜ別案にしなかったか」を辿れるようにする。

## 実データの自取得を標準の使い方とする

利用者が Dukascopy 等から自分で取得した実データ（1h足で十分）で回すのが標準。禁じられているのはデータの塊の再配布であって取得・利用ではないため、リポには取得手順と変換スクリプトを置き、データファイルはコミットしない。同梱の合成データ（シード付き乱数ウォーク）は「テストとデモが誰の環境でも再現できる」ための素材に位置づけを限定する。

## 差し替えは CLI 引数とファイルで行う。対話ウィザードにしない

「インジケーターとパラメータを差し替えて、結果を見て、また回す」がツールの本質価値。パラメータは `--param key=value` の引数上書き（試行のたびに JSON 編集を強制しない）、インジケーターは `IndicatorSet` を公開する Python ファイルの `--indicators` 注入とする。戦略追加は dotfiles 版の「1戦略1ディレクトリ」を踏襲し、examples をコピーして indicators.py と config.json を差し替える方式を README でアドバイスする。

## 動的レジーム切替を採った

静的バックテストは相場環境の変化に弱く共倒れする。相場を短窓で分類し局面ごとに戦略を切り替えることで、過学習を避ける設計判断。比較検証で自分の仮説を潰しにいく姿勢もこの方針の一部（比較の手段は後述の `--json` 汎用化を参照）。

## エッジは枠組みではなく「対応表の中身」。パラメータも対応表も設定 JSON で外部注入する

ADX/ATR/RSI も9セルも動的閾値もクオンツの一般概念で、公開してもエッジは漏れない。唯一エッジ形状を示すのは `regime_strategy`（どの局面で follow/flip が効くか＝答え）。これは中立パラメータでも形が残るため、パラメータだけでなく**対応表自体も外部注入**にし、リポには中立／説明用ダミーのみ置く。線引きは「問いと手法は出す、答えは出さない」。

注入の形式は**設定 JSON 一本**（環境変数 `$BT_DYNAMIC_CONFIG` はファイルパスの指定のみ。初期案にあったパラメータ個別の env 注入は、設定の散逸を招くため廃止した）。本番値をコードに焼き込まず外部注入にすることで、リポにも wheel にもデモにも**そもそも存在しない**状態を作る——守秘を「隠蔽」ではなく「構造的分離」で実現し、再現性（誰でも clone して動かせる）と両立させる。

## 設定キーは指標非依存の一般名にし、ADX/ATR/RSI は「デフォルト指標」に降格する

エンジンが知るのは抽象軸のみ（`ax1` = トレンド強度、`ax2` = ボラティリティ、`direction` = 中心値付き振動子）。閾値キーも `ax1_weak` / `vol_lo` / `direction_band` と一般名にし、ADX/ATR/RSI は `IndicatorSet` の差し替え可能なデフォルト実装に位置づけた。`--indicators file.py` による指標差し替えはこの抽象化が成立条件。

## lot 概念を持たない

エンジンの成績は flat lot の pips 集計のみで、lot 重み付け（旧 compare_lot の adx_prop 等）は持たない。lot サイジングはエンジンの判定ロジックと独立した資金管理の関心事であり、足すならポジションモデルの設計判断が要る。dogfooding で必要が立った時点で Issue として扱う。

## 戦略を trend/ 1本に絞る

「単一仮説を端から端まで」というナラティブが本リポの強み。戦略を足すと「色々試す人」に戻り、エッジ形状のヒントも保守コストも増える。1本をクリーンに見せる方がポートフォリオとして強い。

## データを同梱しない

各データソース（Dukascopy / Alpha Vantage / HistData / yfinance）はデータの塊の再配布を規約で制限する。一方コードの配布は自由。よって「取得コードを配り、データは利用者が自取得」とする。VHS で数行を映すのは復元不能な断片＝ de minimis であり再配布に当たらない。

## Demo で成績を出さない

公開する中立パラメータ × 無料1h足のバックテスト成績は、本番（専用 feed × 秘密パラメータ）と無関係。数字を実績として見せると誤読を生むため、Demo の用途は「仕組みと検証作法の実演」に固定する。
