# Design Decisions

[README](../README.md) の Design Decisions を圧縮せず、判断の全文をここに置く。「なぜそうしたか」だけでなく「なぜ別案にしなかったか」を辿れるようにする。

## 9セルという小さなフレームをあえて固定する

分類は精緻にするほど「正しく」なるが、検証は人間の頭に載らなくなる。セルが増えるほど1セルあたりの標本は薄くなり、比較は表で一瞥できなくなり、どこで何が効いているのかを考えること自体が難しくなる。

そこでベストの分類を探すのをやめ、最初に 3×3 = 9 という小さなフレームを固定し、その制限の下での最適を探すことにした。固定したのは「9 セル × 順張り/逆張り/ノーポジ」という形だけで、軸に使う指標・閾値・対応表はすべて差し替えられる。トレンド強度 × ボラティリティという軸の選び方自体がトレンドフォロー型に寄せたものであり、別のタイプの戦略には別の軸を与えればよい。

固定には運用上の実利もある。分類の境界を一度でも動かすと全セルの定義が変わり、それまで積んだ窓ごとの比較が全部無効になる。だから境界は動かさない。実際、閾値の境界替えが検証候補に挙がったことはあるが、「全セル定義に影響する」を理由に見送っている。フレームを直したくなったときは、直すのではなく同じ軽さの別フレームをもう一つ作る——この粒度の検証器は、作り込むより使い潰す方が価値がある。

## セルは成績でなく生存で選ぶ

フレームが小さいことの見返りは、検証の単位が人間に扱えることにある。季節窓・年窓という誰にでも意味の分かる区切りでセルごとの成績を並べ、複数期間を通してプラスを保ったセルだけを運用対象に昇格させる。単発の好成績は昇格の理由にならず、一つの窓で崩れたパラメータは採らない。

これは「良い成績を探す」のではなく「壊れなかったものだけを残す」手続きであり、過学習への対抗をパラメータ探索の側でなく選抜規則の側に置いている。パラメータ変更も同じ規則に従う。合計を改善する変更でも、どれか一つの窓が崩れるなら戻す。実際に、合計では有利だったパラメータが特定年を崩したため棄却した、という判断を繰り返している。

## 未解決を未解決のまま残す

検証器は答えを出さないことがある。あるセルがどの窓でも成立しない、ある季節にだけ負ける、それまで安定していた成績がある期間から劣化する——こうした結果を、no_trade に確定するでも仮説ごと捨てるでもなく、未解決の台帳として持ち続ける。劣化に対しても「一時的な相場環境の変化か、エッジの構造的縮小か」を決めつけず、判断材料が積もるまで観測を続ける。

これも生存選抜の裏面である。昇格に複数期間の生存を要求する以上、降格や再検証にも同じだけの観測を要求しないと、選抜規則が対称でなくなる。

## 動的レジーム切替を採った

静的バックテスト（過去成績で戦略を固定する）は相場環境の変化に弱く共倒れする。相場を短窓で分類し局面ごとに戦略を切り替えることで、単一戦略の全期間最適化を避ける。分類閾値を直近営業日のパーセンタイルから導出する動的閾値（`--dynamic`）も持ち、静的閾値の恣意性そのものを比較検証の対象にできる。

## エントリーは1本ずらしの次バー open

指標はバー N の確定値で計算し、エントリーはバー N+1 の open で約定させる。先読みバイアスを潰す作法としてバックテストの当然だが、それだけではない。本番では5分足はバーが閉じた瞬間に確定しない。1本ずらしてあるからこそ、本番は確定済みバーで判定して次のバーの頭で発注でき、バックテストの約定条件と一致する。

同じ理由で、セッション終了時の強制決済（EOD）もエンジンが持つ。オーバーナイトの損益を検証に混ぜず、本番と同じ場所で切る。

## エンジンは抽象軸しか知らない

エンジンが知るのは `ax1`（トレンド強度）・`ax2`（ボラティリティ）・`direction`（中心値付き振動子）という抽象軸のみ。閾値キーも `ax1_weak` / `vol_lo` / `direction_band` と一般名にし、ADX/ATR/RSI は `IndicatorSet` の差し替え可能なデフォルト実装に位置づけた。`--indicators file.py` による指標差し替えはこの抽象化が成立条件。

## 設定は JSON 一本の明示ロード

対応表・閾値・パラメータは `Config.load(path)` の明示呼び出しでのみ読む。import 時の暗黙ロードはなく、パラメータ個別の環境変数注入もない（初期案にあったが、設定の散逸を招くため廃止。環境変数 `$BT_DYNAMIC_CONFIG` はファイルパスの指定のみ）。コアは戦略側を import しない一方向依存で、バックテストと本番実行層が同じ config ファイルを読むことで、検証した判定と運用する判定が一致する。

## 差し替えは CLI 引数とファイルで行う。対話ウィザードにしない

「インジケーターとパラメータを差し替えて、結果を見て、また回す」がツールの本質価値。パラメータは `--param key=value` の引数上書き（試行のたびに JSON 編集を強制しない）、インジケーターは `IndicatorSet` を公開する Python ファイルの `--indicators` 注入とする。戦略追加は「1戦略1ディレクトリ」で、examples をコピーして config.json と indicators.py を差し替える。

## 比較スクリプトを持たず `--json` 出力に汎用化する

前身では仮説を比較するたびに専用スクリプトが増え、比較のためのコードが検証対象より速く育った。本リポは機械可読サマリー（`--json`）を吐くだけにし、並べ方は jq なり表計算なり利用者の手に委ねる。仮説が増えてもエンジンは増えない。

## lot 概念を持たない

エンジンの成績は flat lot の pips 集計のみで、lot 重み付けは持たない。lot を判定に混ぜると、判定の良し悪しと資金管理の良し悪しが同じ数字の中に混ざり、どちらが効いたのか切り分けられなくなる。サイジングの検証は、flat lot で判定を固定した上で重み付けだけを別途比較して初めて結論が出る——これはサイジング研究を判定研究と並走させた経験からの結論で、だからエンジンは flat を報告する係に徹し、サイジングは本番実行層の側が持つ。
