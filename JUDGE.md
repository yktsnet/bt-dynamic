## pip（PyPI）配布を採った

module-guide の既定は clone リファレンスだが、`ops_dynamic` が本番でコアを組み込む実利用者としてすでに存在するため「組み込み型 → レジストリ配布必須」に該当する。pip 化は PYTHONPATH 参照をバージョン固定依存に置き換える実運用の改善であり、かつ「wheel に答えが存在しない」ことが構造的分離の証明になる。versioning・後方互換・英語 doc の継続コストは承知の上で払う。名前は `bt-dynamic` / `bt_dynamic` に統一（`dynamic-regime-bt` 案は破棄）。

## 動的レジーム切替を採った

静的バックテストは相場環境の変化に弱く共倒れする。相場を短窓で分類し局面ごとに戦略を切り替えることで、過学習を避ける設計判断。比較スクリプト（dynamic / regime / lot）で自分の仮説を潰しにいく姿勢もこの方針の一部。

## パラメータを Env で外出しする

守秘（本番のエッジ）と再現性（誰でも clone して動かせる）を両立させる手段。
本番パラメータをコードに焼き込まず外部注入にすることで、リポにもデモにも**そもそも存在しない**状態を作る。守秘を「隠蔽」ではなく「構造的分離」で実現する。

## エッジは枠組みではなく「対応表の中身」。対応表も外部注入する

ADX/ATR/RSI も9セルも動的閾値もクオンツの一般概念で、公開してもエッジは漏れない。
唯一エッジ形状を示すのは `REGIME_STRATEGY`（どの局面で follow/flip が効くか＝答え）。これは中立パラメータでも形が残るため、パラメータだけでなく**対応表自体も Env/設定で外部注入**し、リポには中立／説明用ダミーのみ置く。線引きは「問いと手法は出す、答えは出さない」。

## 戦略を trend/ 1本に絞る

「単一仮説を端から端まで」のナラティブが本リポの強み。戦略を足すと「色々試す人」に戻り、エッジ形状のヒントと保守コストが増える。1本をクリーンに見せる方がポートフォリオとして強い。

## README / Zenn に投資する

リポの価値は配管コードでなく思考。文章化して初めて汎用エンジンとの差（完結性）が立つ。書くのは問いと手法まで。成績・対応表の中身・どのセルが効いたかは書かない。「答えは構造的に分離してある」と言えること自体が設計力の証明になる。

## データを同梱しない

各データソース（Dukascopy / Alpha Vantage / HistData / yfinance）はデータの塊の再配布を規約で制限する。一方コードの配布は自由。よって「取得コードを配り、データは利用者が自取得」とする。
VFS／VHS で数行を映すのは復元不能な断片＝ de minimis であり再配布に当たらない。

## Demo で成績を出さない

公開する中立パラメータ × 無料1h足のバックテスト成績は、本番（専用 feed × 秘密パラメータ）と無関係。
数字を実績として見せると誤読を生むため、Demo の用途は「仕組みと検証作法の実演」に固定する。
